# 东方超市菜谱搜索引擎 — 设计文档

- **日期**：2026-06-14
- **项目**：eastern-recipe（独立 GitHub Pages 静态站，沿用 eastern-farm 套路）
- **状态**：已通过方向评审，待写实施计划

## 1. 目标与定位

把东方超市的**进货 + 销售数据**变成顾客的"想吃什么 → 我们这都有"的入口，同时反过来帮店里清库存。两个一体的用途：

- **A. 顾客端引流**：顾客搜一道菜（如"麻婆豆腐"），系统给出菜谱 + 所需食材清单，每样标注东方超市的价格 / 分类货架 / 是否在售，引导顾客"我想做这道菜"→"来东方买齐"。
- **B. 库存反推促销**：根据当季 / 在售 / 店主手动主推的食材，反查"用到这些食材、且其余配料店里也都有"的菜谱，排序推荐，可导出成促销海报推给会员、清库存、提客单价。

**非目标（YAGNI，v1 不做）**：会员登录、个性化、购物车结算、实时库存数量显示、AI 实时生成菜谱。

## 2. 关键决策与约束

- **菜谱来源 = 自建精选库（手工绑定 SKU）**。准确性优先：每道菜的每样食材都绑定到真实在售商品，保证引流精准、可反推促销。搜不到的菜暂不兜底（未来可加 AI 兜底，非 v1）。
- **平台 = 独立静态网页**（vanilla JS，无 build，GitHub Pages 托管），与 eastern-farm 同构。顾客端不暴露 StockWise 后台。
- **⚠️ 库存数量不可信**：StockWise `products.stock_quantity` 已知不准。因此本项目**不显示"还剩几份"**，只绑定可靠字段：商品身份（itemCode）、价格、单位、分类。B 的"可做"判断基于"该商品是否在售 + 店主主推标记"，不依赖库存数字。
- **数据锚点 = Clover 销售明细**：Clover item 名称本身是中英双语（"Tong Ho Crown Daisy 茼蒿"、"Cilantro 芫荽"），带 `itemCode`、`price`、`unitName`。这是天然的"食材→SKU"绑定基础。

## 3. 架构总览

```
Clover 商品数据 ──(每日导出脚本)──▶ products.json ─┐
                                                  ├─▶ 静态网页（vanilla JS）
店主手工精选菜谱 ──(半自动绑定脚本)──▶ recipes.json ┘      ├─ A: 搜菜找货
                                                          └─ B: 今天做什么菜
```

三个隔离的单元，各自单一职责、接口清晰、可独立理解和测试：

1. **导出器（export script）** — 从 Clover/StockWise 拉在售商品，产出 `products.json`。
2. **绑定器（binding script）** — 半自动把菜谱食材匹配到 Clover 商品名，产出候选绑定供店主确认，沉淀进 `recipes.json`。
3. **前端站点（static site）** — 读两个 JSON，提供 A / B 两个体验。

## 4. 数据模型

### products.json（每日自动导出，机器生成，勿手改）
```json
{
  "generated_at": "2026-06-14T21:00:00-06:00",
  "items": [
    {
      "itemCode": "10588",
      "name_cn": "芫荽",
      "name_en": "Cilantro",
      "price": 0.99,
      "unit": "each",
      "category": "蔬菜",
      "on_sale": true
    }
  ]
}
```
- `itemCode` 为主键（来自 Clover `item.code`）。
- `on_sale` = 该商品当前在 Clover 在售（available 且未 deleted），**不代表具体剩余数量**。

### recipes.json（手工精选 + 半自动绑定，人审）
```json
{
  "recipes": [
    {
      "id": "mapo-tofu",
      "name_cn": "麻婆豆腐",
      "name_en": "Mapo Tofu",
      "aliases": ["麻婆豆腐", "mapo", "麻辣豆腐"],
      "image": "images/mapo-tofu.jpg",
      "tags": ["川菜", "下饭", "快手"],
      "steps": ["豆腐切块焯水", "..."],
      "ingredients": [
        { "label": "嫩豆腐", "qty": "1 盒", "itemCode": "4011tofu", "required": true },
        { "label": "郫县豆瓣酱", "qty": "2 勺", "itemCode": "ddbanjiang", "required": true },
        { "label": "牛肉末", "qty": "200g", "itemCode": null, "required": false }
      ]
    }
  ]
}
```
- `itemCode: null` = 暂未绑定 / 店里没有对应商品（前端标"暂缺"，不阻断）。
- `required` = 是否核心食材；B 的"可做"判断只看 `required` 的食材是否都在售。

### 关联逻辑（前端启动时）
对每道菜的每个 ingredient，用 `itemCode` 去 `products.json` 查：在售→带出价格/分类；未在售或 `itemCode:null`→标"暂缺"。

## 5. 功能流程

### A. 顾客搜菜（引流）
1. 顾客输入菜名（支持中英文、`aliases` 模糊匹配）。
2. 命中菜谱 → 展示图 + 标签 + 步骤 + 食材清单。
3. 每样食材一行：`【食材名 · 用量】` → `【¥价格 · 分类】` 或 `【暂缺】`。
4. 底部汇总：「这道菜共需 ¥XX，X/Y 样我们都有」+「生成购物清单」（可截图/二维码到店找货）。

### B. 今天做什么菜（清库存/促销）
1. 输入 = 当季在售食材 + 店主在后台标的"主推食材"（一个简单清单文件，店主手填）。
2. 反查：所有 `required` 食材都在售、且至少用到 ≥1 个主推食材的菜谱。
3. 按"用到主推食材数量 / 食材齐全度"排序推荐。
4. 选中可一键导出促销海报素材（菜名 + 图 + 主推食材 + 价格），接入现有会员推送流程。

## 6. 绑定数据怎么攒（解决 B 方案的主要成本）

半自动 `binding script`：
1. 维护一份常见家常菜食材词表（含别名，如"豆瓣酱"="郫县豆瓣"="豆瓣"）。
2. 拿食材名对 `products.json` 商品名做中英双语模糊匹配，生成"候选绑定"（食材 → top-N 候选 itemCode + 置信度）。
3. 店主过一遍：高置信直接采纳，低置信人工选或标"暂缺"。
4. 结果写回 `recipes.json`。新品上架时重跑、补绑。

首批目标：50–100 道东方超市顾客最常做的家常菜。

## 7. 错误处理与边界

- 搜不到菜：提示"暂无此菜谱"，并记录搜索词（沉淀下一批要补的菜谱），**不报错、不兜底生成**（v1）。
- `products.json` 拉取失败：前端用上一次缓存的 JSON，页面顶部提示"价格更新于 X"。
- 食材未绑定/暂缺：正常展示菜谱，该食材标"暂缺"，不影响其他食材引流。
- 中文/微信浏览器兼容：沿用 eastern-farm 已踩过的坑（无 build、纯静态、注意微信内置浏览器限制）。

## 8. 测试策略

- **导出器**：对样例 Clover 数据跑，断言 `products.json` 字段完整、itemCode 唯一、价格为数值。
- **绑定器**：给定食材词 + 商品列表，断言匹配候选与置信度排序符合预期（含中英双语、别名用例）。
- **前端关联**：给定 products + recipes 夹具，断言每道菜的"齐全度""共需金额""暂缺标记"计算正确。
- **B 排序**：给定主推食材集合，断言推荐集合与排序正确（含"required 缺货则排除"用例）。

## 9. 分期

- **v1（本 spec）**：方案 1 静态站 + A + B + 首批 50–100 菜谱 + 每日导出 + 半自动绑定。
- **v2（未来，另开 spec）**：升级方案 3（React+Firebase 复用 eastern-market-members），加会员登录、保存购物清单、AI 兜底生成搜不到的菜。
